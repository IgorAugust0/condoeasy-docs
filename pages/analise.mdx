# Análise

A seção de Análise tem como objetivo transformar os dados coletados em uma ideia concreta para a resolução do problema. Nesta fase, ocorre o levantamento e a análise de requisitos, que marcam o ponto de partida para o desenvolvimento de software. O principal foco é entender e capturar as necessidades e expectativas do cliente ou usuário final.

## Atividades de Análise

As principais atividades desta fase incluem o levantamento, análise, especificação e validação dos requisitos do sistema. Estas atividades são formalizadas em documentos que orientam o desenvolvimento de acordo com as necessidades do projeto. Entre os principais artefatos gerados nesta etapa, destacam-se:

- **Documento de Requisitos**
- **Especificação de Casos de Uso**
- **Matriz de Rastreabilidade de Requisitos**

### Documento de Requisitos

O **Documento de Requisitos** é uma declaração oficial que descreve o que os desenvolvedores devem implementar no sistema. Ele pode conter diferentes tipos de requisitos:

- **Requisitos de Usuário**: descrevem o que os usuários esperam do sistema.
- **Requisitos de Sistema**: definem as especificações técnicas que o sistema deve atender.
- **Requisitos Funcionais**: tratam das funcionalidades que o sistema deve oferecer.
- **Requisitos Não-Funcionais**: abordam características como desempenho, segurança e usabilidade.

Este documento é utilizado por vários usuários e sua estrutura pode variar de acordo com o tipo de software e a abordagem de desenvolvimento. Uma organização comum, sugerida por Sommerville e baseada na norma IEEE, ajuda a evitar problemas e garantir uma documentação clara e eficiente.

### Matriz de Rastreabilidade de Requisitos

A **Matriz de Rastreabilidade de Requisitos** é uma ferramenta que estabelece a relação entre os requisitos e outros componentes do projeto. Ela permite:

- Rastrear a origem de cada requisito.
- Identificar como os requisitos se conectam com outros elementos do projeto.
- Detectar inconsistências nos requisitos.
- Garantir que os diferentes níveis do projeto estejam alinhados.

Essa matriz é essencial para gerenciar mudanças, pois facilita a identificação de quais requisitos serão afetados caso uma alteração seja feita no projeto.

### Especificações de Casos de Uso

As **Especificações de Casos de Uso** são documentos funcionais que descrevem as funcionalidades que o sistema deve ter, bem como o comportamento esperado. Eles detalham como o sistema deve reagir a diferentes entradas e quais saídas devem ser geradas, envolvendo operações como cálculos e manipulação de dados. 

- Esses documentos oferecem uma visão clara do que o sistema fará e como será utilizado, garantindo que os requisitos sejam bem definidos e as expectativas dos usuários atendidas.
- Além disso, os casos de uso são úteis no planejamento de testes de aceitação, assegurando a conformidade do sistema com os requisitos do usuário.

### Protótipo de Baixa Fidelidade

O **Protótipo de Baixa Fidelidade** é uma representação visual preliminar do sistema, focado em testar ideias e validar conceitos sem se preocupar com detalhes gráficos ou funcionais. Ele é utilizado para explorar as interações básicas e a estrutura da interface, permitindo feedback rápido de usuários e partes interessadas sobre o design e usabilidade antes de se avançar para protótipos mais detalhados.
